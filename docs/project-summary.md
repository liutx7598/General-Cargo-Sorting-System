## 2. 项目目标

本项目是一个面向件杂货船/杂货船的“智能配载系统”MVP 原型，目标是形成一条完整、可运行、可继续扩展的业务链路：

1. 管理船舶、货舱、货物、航次、规则、告警等基础数据
2. 发起配载任务
3. 调用算法服务执行分舱、摆位、重心计算、GM 核算和规则校验
4. 返回 PASS / FAIL 结论与原因
5. 在前端以指标、表格、二维总配载图和三维货舱视图展示结果

当前实现坚持以下原则：

- 参数化适配船型
- 算法与业务解耦
- 前后端分离
- 为后续接入 LLM、压载、剪力/弯矩等能力预留接口

## 7. 算法设计与我们已经完成的优化

### 7.1 原始求解流程

算法服务按照四阶段设计：

1. 阶段 A：静态筛选
2. 阶段 B：货舱分配
3. 阶段 C：舱内摆位
4. 阶段 D：核算与回退

### 7.2 已落地的求解策略

#### 阶段 A：静态筛选

已经实现：

- 过滤尺寸放不下的货舱
- 过滤重量超限货舱
- 为每件货物生成 `candidateHolds`

#### 阶段 B：货舱分配

当前实现为：

- 启发式初分配
- OR-Tools CP-SAT 精修

并且已经针对件杂货场景做了重要优化：

##### 优化 1：优先少开舱

过去的分舱策略偏向“平衡利用率”，会把货物摊到多个货舱。  
这与件杂货现场习惯不一致。

现在已经改为：

- 优先最少开舱
- 优先使用中部优先级更高的货舱
- 已开启货舱优先继续装
- 当前舱确实装不下时才开下一舱

这部分逻辑位于：

- `algorithm-service/app/solver/hold_allocator.py`

#### 阶段 C：舱内摆位

当前实现为：

- 分层
- 分排
- Best-Fit Decreasing 风格的启发式摆位

并且已经针对件杂货“成舱、成排、成块”的场景继续优化：

##### 优化 2：优先同层聚拢

用户明确提出：

- 件杂货通常先装满一个舱，再考虑下一个舱
- 同一层未铺满之前，应尽量把货物放在一起

基于这个要求，我们已经做了以下优化：

- 优先放在同一舱
- 优先在同一层开新排
- 降低“为了节省长度而把货物立起来”的倾向
- 对高度膨胀的摆放朝向增加惩罚
- 优先低矮、稳定、平放的摆放方式

这部分逻辑位于：

- `algorithm-service/app/solver/packer.py`

##### 优化结果

在当前示例任务中，货物已从原来的“分散在多个舱”优化为：

- 集中装在单一货舱
- 全部保留在第 1 层
- 采用多排平铺，而不是把后续货物竖直摆放

#### 阶段 D：回退修复

已实现：

- 对未成功摆放货物进行换舱修复
- 修复后重新执行摆位
- 最多迭代若干次

同时修复了一个关键问题：

- 之前回退后可能再次被分配器改写
- 现在修复结果会被直接保留，不再被旧分配结果覆盖

### 7.3 规则校验改进

我们还修复了隔离规则的一处关键逻辑：

- 之前 `isolationLevel > 0` 可能错误地作用于所有货物对
- 现在改为仅对：
  - 明确不兼容货物对
  - 或危险货对
  应用隔离距离

这让普通货与木材等非禁配货物可以正常同舱装载，避免误报。

## 11. 已解决的联调问题

截至目前，我们已经解决了以下关键问题：

### 11.1 前端点击无反应 / Network Error

原因：

- 前端代理端口错误
- 后端或算法服务未启动

解决：

- 修正 Vite 代理
- 增加错误提示
- 提供一键启动脚本

### 11.2 唯一键冲突

原因：

- 示例数据与前端默认编号重复

解决：

- 生成唯一编码
- 修正数据库序列
- 优化错误提示

### 11.3 货物分散到多个舱

原因：

- 分舱目标偏向均衡利用率

解决：

- 改为优先最少开舱、已开舱优先继续装

### 11.4 同层未铺满却把货立起来

原因：

- 舱内摆位过于偏向复用当前货排

解决：

- 增加对高度膨胀的惩罚
- 优先同层新开货排
- 优先低矮稳定摆放

### 11.5 快捷复制货物报错

原因：

- 页面调用了新方法，但浏览器热更新中仍保留旧版 store

解决：

- 页面增加回退逻辑
- 即使批量保存方法不存在，也退回逐条保存
